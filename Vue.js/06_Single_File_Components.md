# 06_Vue3_Single_File_Components (SFC)

## 1. Component (컴포넌트)

### 컴포넌트란?
- 웹 페이지를 구성하는 **재사용 가능한 코드 블록**이다.
- 화면의 UI를 마치 '레고 블록'처럼 독립적인 기능을 가지는 조각들로 분할하여 개발한다.
- 이 블록들을 조립하여 전체 페이지를 만들기 때문에, 코드의 **재사용성이 높아지고 유지보수가 매우 쉬워진다.**

### 컴포넌트의 특징
- UI(User Interface: 사용자가 마주하는 화면의 시각적 디자인)를 독립적이고 재사용 가능한 일부분으로 분할하고, 각 부분을 개별적으로 다룰 수 있다.
- 자연스럽게 애플리케이션은 **중첩된 컴포넌트의 트리(Tree) 형태**로 구성된다.
  - 예: `<Root>` 밑에 `<Header>`, `<Main>`, `<Aside>`가 있고, `<Main>` 밑에 여러 개의 `<Article>`과 `<Item>`이 존재하는 구조.
- 실제 우리가 사용하는 페이스북, 네이버 VIBE 등 대부분의 웹 서비스는 여러 개의 컴포넌트로 이루어져 있다.

---

## 2. Single-File Components (SFC)

### SFC 란?
- 컴포넌트의 템플릿(HTML), 로직(JavaScript), 그리고 스타일(CSS)을 **하나의 파일로 묶어낸 특수한 파일 형식 (`*.vue` 파일)** 이다.
- 관련된 코드가 한곳에 모여 있어 컴포넌트 단위의 개발과 유지보수가 매우 편리하고 체계적으로 이루어진다.

### 컴포넌트 사용 및 빌드(Build)
- Vue SFC 코드는 일반적인 HTML 파일처럼 브라우저에서 바로 실행할 수 없다.
- 컴파일러를 통해 개발자가 쓴 소스 코드를 브라우저가 실행 가능한 파일(표준 HTML, JS, CSS)로 변환하는 **빌드(Build)** 과정이 반드시 필요하다.
- 실제 프로젝트에서는 **Vite**와 같은 공식 빌드 도구를 사용하여 프로젝트를 구성한다.
- 간단한 컴포넌트 코드 작성 및 미리보기는 공식 홈페이지의 [Vue SFC Playground (play.vuejs.org)](https://play.vuejs.org/)에서 테스트해 볼 수 있다.

## 3. SFC 파일 구성 요소

각 `*.vue` 파일은 세 가지 유형의 최상위 언어 블록인 `<template>`, `<script>`, `<style>`로 구성된다. 
(언어 블록의 작성 순서는 상관없으나, 일반적으로 `template` $\rightarrow$ `script` $\rightarrow$ `style` 순서로 작성한다.)

```vue
<!-- SFC 구조 예시 -->
<template>
  <div class="greeting">{{ msg }}</div>
</template>

<script setup>
import { ref } from 'vue'
const msg = ref('Hello World!')
</script>

<style scoped>
.greeting {
  color: red;
}
</style>
```

### 1) `<template>` 블록
- 화면의 구조를 담당하는 HTML 코드를 작성하는 영역이다.
- 각 `*.vue` 파일은 **최상위 `<template>` 블록을 하나만 포함**할 수 있다.

### 2) `<script setup>` 블록
- 컴포넌트의 로직을 담당하는 JavaScript 코드를 작성하는 영역이다.
- 각 `*.vue` 파일은 `<script setup>` 블록을 **하나만 포함**할 수 있다. (일반 `<script>` 블록 제외)
- 컴포넌트의 `setup()` 함수로 사용되며, 컴포넌트의 각 인스턴스가 생성될 때마다 실행된다.
- **💡특징:** `<script setup>` 내에 선언된 **변수 및 함수는 동일한 컴포넌트의 템플릿(`<template>`)에서 별도의 반환(return) 없이 자동으로 사용 가능**하다.

### 3) `<style scoped>` 블록
- 컴포넌트의 스타일을 담당하는 CSS 코드를 작성하는 영역이다.
- `*.vue` 파일에는 **여러 개의 `<style>` 태그가 포함될 수 있다.**
- **`scoped` 속성:** 이 속성이 지정되면, 작성된 CSS는 **현재 컴포넌트의 요소에만 적용**된다. (다른 컴포넌트의 디자인에 영향을 주지 않음)

## 4. Vue 컴포넌트 활용

컴포넌트를 실제 프로젝트에서 사용하는 과정은 크게 3단계로 이루어진다.

### 컴포넌트 사용 3단계
**1. 사전 준비**
- 초기에 생성된 기존 컴포넌트들을 모두 삭제하고, 최상위 컴포넌트인 `App.vue`의 코드를 초기화하여 비워둔다.

**2. 컴포넌트 파일 생성**
- `src/components` 폴더 하위에 새로운 컴포넌트 파일(예: `MyComponent.vue`)을 생성하고, `<template>`과 `<script setup>` 블록을 작성한다.

**3. 컴포넌트 등록 (import)**
- 부모 컴포넌트(예: `App.vue`)의 `<script setup>` 안에서 생성한 자식 컴포넌트를 `import` 한다.
- `import` 한 컴포넌트는 `<template>` 내에서 HTML 태그처럼 `<MyComponent />` 형식으로 사용할 수 있다.
- **경로 지정 팁:** 경로 작성 시 `@` 기호는 `src/` 디렉토리를 뜻하는 약어이다. (예: `@/components/MyComponent.vue`)

> **결과 확인:** Vue DevTools를 활용하면 브라우저 환경에서 `App` 컴포넌트 하위에 `MyComponent`가 정상적으로 종속(부모-자식 관계)되어 있는지 시각적으로 확인할 수 있다. 추가로 `MyComponent` 하위에 `MyComponentItem`을 만들어 컴포넌트의 재사용성과 중첩 구조를 확장할 수 있다.

## 5. SFC 빌드 도구 (Build Tool)

웹 브라우저는 표준 HTML, CSS, JavaScript만 이해할 수 있으므로, `*.vue` 파일과 같은 최신 문법 및 프레임워크 코드를 브라우저가 인식할 수 있는 형태로 변환해 주는 도구가 필요하다.

### Vite (바이트)
- Vue 프론트엔드 개발을 위한 빠르고 쾌적한 **빌드(Build) 도구**이자 개발 서버이다.
- 개발 서버 시작 속도가 매우 빠르고, 코드 수정 시 실시간에 가까운 즉각적인 반영(HMR)을 제공한다.

### 빌드(Build)와 번들링(Bundling)
- **빌드(Build):** 프로젝트의 소스 코드를 최적화하고 번들링하여 실제 서버에 배포할 수 있는 형식으로 변환하는 전체 과정이다.
- **번들링(Bundling):** 개발 과정에서 여러 개로 흩어져 작성된 코드 파일(JS, CSS, 이미지 등)들을 최적화하여 하나(혹은 소수)의 파일로 합쳐주는 작업이다.

### Node.js와 NPM
- **Node.js:** 브라우저 안에서만 동작하던 JavaScript를 브라우저가 아닌 서버 측(Server-Side)이나 로컬 PC에서도 실행할 수 있게 해주는 실행 환경(Runtime)이다. 이를 통해 프론트엔드 개발 도구들을 로컬에서 구동할 수 있다.
- **NPM (Node Package Manager):** 전 세계 개발자들이 만든 다양한 JavaScript 패키지(라이브러리)들을 모아 놓은 저장소이자, 이를 명령어(예: `npm install`) 하나로 쉽게 설치하고 관리할 수 있게 해주는 도구이다.

## 6. Vue 프로젝트 생성 실습

Vite를 기반으로 하는 공식 Vue 프로젝트 생성 과정이다.

**1. 프로젝트 생성 명령어 실행**
- 터미널에 아래 명령어를 입력한다. (`latest`는 최신 안정화 버전을 사용하겠다는 의미이다.)
```bash
$ npm create vue@latest
```

**2. 프로젝트 설정**
- **Project name:** 생성할 프로젝트 폴더의 이름을 입력한다. (예: `vue-project`)
- 이후 나타나는 추가 설정(TypeScript, Vue Router, Pinia 등)은 방향키와 `Space`로 선택하며, `Enter`를 눌러 기본값(No)으로 설정 후 넘어갈 수 있다.

**3. 프로젝트 실행 준비 및 서버 구동**
- 안내되는 명령어 순서에 따라 프로젝트 폴더로 이동 후 패키지를 설치하고 서버를 실행한다.
```bash
$ cd vue-project   # 1. 생성된 프로젝트 폴더로 이동
$ npm install      # 2. 프로젝트 실행에 필요한 패키지들 일괄 다운로드 (인터넷 연결 필수)
$ npm run dev      # 3. 개발 서버 실행
```
- 실행 후 터미널에 출력된 `Local: http://localhost:5173/` 링크를 `Ctrl + 클릭`하여 브라우저에서 렌더링 된 초기 화면을 확인한다.

## 7. 모듈(Module)과 번들러(Bundler)

### 모듈 (Module)
- 애플리케이션의 크기가 커지면서, 파일 하나에 모든 코드를 담기 어려워 파일을 여러 개로 분리하게 되는데, 이 **분리된 각 파일(`*.js`)** 을 모듈이라고 한다.
- **모듈의 한계:** 모듈의 개수가 극적으로 증가하면 네트워크 요청이 많아져 성능 병목 현상이 발생하고, 모듈 간의 의존성(연결성)이 깊어져 관리가 매우 어려워진다. (`node_modules`의 블랙홀 현상)

### 번들러 (Bundler)
- 복잡한 모듈 간의 의존성 문제를 해결하기 위한 도구이다.
- 여러 모듈과 파일을 하나의 번들로 **묶고 최적화**하여 브라우저의 네트워크 요청 부담을 줄여준다.
- 웹사이트의 로딩 속도를 높이고 성능을 최적화하는 핵심 역할을 한다.
- **TIP:** Vite는 내부적으로 `Rollup`이라는 번들러를 사용하며, 개발자가 복잡한 설정 없이 바로 사용할 수 있도록 기본 환경이 세팅되어 있다. 또한, import하지 않은 코드를 자동으로 제거하는 '트리 쉐이킹(Tree-shaking)' 기능도 수행한다.

## 8. Vue 프로젝트 기본 구조

생성된 Vue 프로젝트의 주요 디렉토리와 파일들의 역할이다.

**1. `public/` 디렉토리**
- 번들러를 거치지 않고 그대로 제공되는 **정적 파일**들을 위치시킨다. (예: `favicon.ico`)
- 소스코드에서 `import` 할 필요가 없으며, 항상 최상위 root(`/`) 절대 경로를 사용하여 참조한다.

**2. `src/` 디렉토리 (핵심 영역)**
- 실제로 개발자가 작성하고 작업하는 대부분의 **소스 코드**가 위치하는 곳이다.
- **`src/assets/`**: 프로젝트 내에서 사용되며 번들러를 거치는 정적 자원(CSS, 이미지, 폰트 등)을 관리한다.
- **`src/components/`**: 애플리케이션에서 재사용 가능한 하위 컴포넌트(`*.vue`)들을 모아두는 곳이다.
- **`src/App.vue`**: Vue 앱의 **Root(최상위) 컴포넌트**이다. 다른 하위 컴포넌트들을 포함하며, 앱 전체의 레이아웃과 공통 요소를 정의한다.
- **`src/main.js`**: 앱의 **진입점(Entry Point)** 이다. Vue 인스턴스를 초기화하고, `App.vue` 컴포넌트를 브라우저의 DOM(`index.html`)에 마운트(연결)하는 역할을 수행한다. 전역 라이브러리 설정도 이곳에서 한다.

**3. `index.html`**
- Vue 앱의 기본이자 유일한 HTML 파일이다.
- `main.js`에 의해 `App.vue`의 내용이 이 HTML 파일의 특정 `<div>`에 마운트되어 화면에 보여진다. (SPA - Single Page Application 구조)

**4. 기타 설정 파일**
- `jsconfig.json`: 컴파일 옵션 및 모듈 시스템 등 경로 관련 설정을 담당한다.
- `vite.config.js`: Vite 빌드 툴에 대한 설정 파일이다. (플러그인, 빌드 옵션, 개발 서버 포트 설정 등)

## 9. 패키지 관리 도구 파일

NPM을 통해 관리되는 패키지(라이브러리)들의 정보와 실제 파일들이 저장되는 영역이다. 역할에 따라 명확히 구분된다.

### 1) `package.json` (설계도)
- 프로젝트에 관한 기본 정보와 프로젝트가 의존하고 있는 패키지들의 목록 및 버전 범위를 정의하는 **메타데이터 파일**이다.
- `npm install` 명령어를 실행하면 NPM은 이 파일의 `dependencies` 항목을 읽고 필요한 패키지들을 찾아 설치한다.
- **비유:** 집을 짓기 전 필요한 재료의 목록과 대략적인 규격이 적힌 **"건축 계획서(설계도)"** 이다.

### 2) `package-lock.json` (상세 내역서 / 영수증)
- `package.json`을 기반으로 **실제 설치된 패키지들의 "정확한 버전 정보"와 의존성 트리**를 촘촘하게 기록하는 파일이다.
- 협업 시 모든 개발자가 혹은 배포 환경에서 완전히 동일한 패키지 버전을 설치할 수 있도록 안정성을 보장해 준다.
- `npm install`을 할 때마다 자동으로 업데이트되므로 개발자가 직접 수정하지 않는다.
- **비유:** 장바구니에 담긴 물건들의 정확한 브랜드, 생산일자, 스펙이 모두 적힌 **"구매 내역서(영수증)"** 이다.

### 3) `node_modules` (자재 창고)
- `npm install`을 통해 다운로드된 패키지의 **실제 소스 코드 파일(물리적 파일)들이 저장되는 디렉토리**이다.
- 용량이 매우 크고 방대해질 수 있으므로, Git으로 버전 관리를 할 때는 추적하지 않도록 `.gitignore` 파일에 반드시 포함되어야 한다. (코드를 공유받은 사람은 `package.json`만 있으면 언제든 `npm install`로 똑같이 재현할 수 있기 때문)
- **비유:** 설계도와 내역서를 바탕으로 실제로 물건(패키지)들이 배송되어 쌓여 있는 **"자재 창고"** 이다.
